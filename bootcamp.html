How the interview actually works

Most analytics technical screens are 3–6 SQL questions in 30–60 minutes, usually on a shared editor (CoderPad, HackerRank) or an async take-home with a messy dataset. There's almost always one scenario problem ("top performer per segment", "month-over-month growth"). The take-home case study is often the deciding round.

They don't just check "does it run." They grade:

  • Correctness under edge cases — did you handle NULLs, duplicates, and ties?
  • Readability — CTEs over deeply-nested subqueries; clear aliases.
  • Decomposition — breaking a hard question into steps out loud.
  • Business sense — asking clarifying questions ("does a lead count at 'demo' or 'won'?").
💡 Three habits to build from Day 1: (1) narrate your logic out loud, (2) ask "what's the edge case — NULLs? dupes? ties?" before calling a query done, (3) write it clean the first time.

Set this up once (Day 1)

You need a database to run queries against all 10 days. Easiest: DB Fiddle (pick PostgreSQL 15, nothing to install). Keep a scratch tab open the whole bootcamp.

DAY 1

Get querying — foundations

~3.5 hrs

SELECT · WHERE · ORDER BY · LIMIT · DISTINCT — and a working practice database.

What you're learning

Every query is SELECT columns FROM table plus optional clauses. The clauses are always written in this order — and it's worth knowing SQL logically executes them in a different order: FROM → WHERE → GROUP BY → SELECT → ORDER BY → LIMIT. That's why you can't use a column alias from SELECT inside WHERE (WHERE runs first).

WHERE filters rows with comparisons (=, <>, >, <, >=, <=) combined with AND/OR/NOT. Text goes in single quotes. ORDER BY col DESC sorts; LIMIT n caps rows (together they give "top N"). DISTINCT collapses duplicate rows.

Key syntax

-- top 5 highest-budget Paid Search campaigns
SELECT name, budget
FROM   campaigns
WHERE  channel = 'Paid Search' AND budget > 5000
ORDER BY budget DESC
LIMIT  5;

-- the distinct set of channels you run
SELECT DISTINCT channel FROM campaigns;

DAY 2

Filter with precision — NULLs, strings, dates

~4 hrs

IN · BETWEEN · LIKE · NULL handling · string & date functions.

What you're learning

IN ('US','UK') beats a chain of ORs; BETWEEN a AND b is inclusive; LIKE '%Q3%' pattern-matches (% = any characters, _ = one). The big one: NULL is "unknown", and comparisons to it return unknown, not true. So WHERE col = NULL silently matches nothing — you must use IS NULL / IS NOT NULL. COALESCE(a, b, 0) returns the first non-null; NULLIF(a,b) returns NULL when equal (handy to dodge divide-by-zero).

String tools: CONCAT, SUBSTRING, TRIM, UPPER/LOWER, REPLACE. Date tools: EXTRACT(YEAR FROM d), DATE_TRUNC('month', d), and date math.

Key syntax

-- leads missing a company size (a real data-quality check)
SELECT * FROM leads WHERE company_size IS NULL;

-- safe division with NULLIF, default with COALESCE
SELECT COALESCE(revenue, 0) / NULLIF(orders, 0) AS avg_order_value
FROM accounts;
⚠️ = NULL never works, and <> excludes NULLs too. If you want "not 'won', including unknowns," you need status <> 'won' OR status IS NULL. Interviewers plant this.

DAY 3

Crunch the numbers — aggregation

~4 hrs

GROUP BY · HAVING · COUNT/SUM/AVG · conditional aggregation.

What you're learning

GROUP BY collapses rows into one per group; aggregate functions (COUNT, SUM, AVG, MIN, MAX) summarize each group. Rule: every non-aggregated column in SELECT must be in GROUP BY. COUNT(*) counts rows; COUNT(col) ignores NULLs; COUNT(DISTINCT col) counts unique values.

WHERE vs HAVING: WHERE filters rows before grouping; HAVING filters groups after. The power move is conditional aggregation — SUM(CASE WHEN ... THEN 1 ELSE 0 END) — to pivot categories into columns in one pass.

Key syntax

-- leads and conversion per channel, in one pass
SELECT c.channel,
       COUNT(*)                                          AS leads,
       SUM(CASE WHEN l.status = 'won' THEN 1 ELSE 0 END) AS won,
       ROUND(100.0 * SUM(CASE WHEN l.status='won' THEN 1 ELSE 0 END) / COUNT(*), 1) AS win_pct
FROM leads l JOIN campaigns c ON c.campaign_id = l.campaign_id
GROUP BY c.channel
HAVING COUNT(*) >= 5
ORDER BY win_pct DESC;

DAY 4

Master the join

~4.5 hrs

INNER / LEFT / RIGHT / FULL · self & anti-joins · the fan-out trap.

What you're learning

INNER JOIN keeps only matches; LEFT JOIN keeps all left rows (NULLs where no match) — this is what you want for "all customers, even those with zero orders". A self-join joins a table to itself (employee → manager). An anti-join finds rows with no match: LEFT JOIN ... WHERE right.id IS NULL (or NOT EXISTS).

The trap they love: fan-out. If you join a table to another with multiple matching rows, the left rows multiply, and any SUM over them double-counts. Always ask "is this join one-to-one or one-to-many?" Aggregate to the right grain first, or use COUNT(DISTINCT).

Key syntax

-- campaigns that produced ZERO leads (anti-join)
SELECT c.name
FROM campaigns c
LEFT JOIN leads l ON l.campaign_id = c.campaign_id
WHERE l.lead_id IS NULL;
⚠️ Put match conditions in ON, filters in WHERE — but note: a filter on the right table's column in WHERE quietly turns a LEFT JOIN back into an INNER JOIN. Keep those conditions in the ON clause.

DAY 5

Break it into CTEs

~4 hrs

Subqueries · CTEs · chaining · recursive CTEs.

What you're learning

A subquery is a query inside a query. Correlated subqueries reference the outer row (run per-row, e.g. "leads above their channel's average"); uncorrelated ones run once. CTEs (WITH name AS (...)) name a subquery so you can chain readable steps instead of nesting. Recursive CTEs walk hierarchies (org charts, category trees).

The interview lesson: reach for CTEs to make hard queries legible. A three-step CTE that a reviewer can follow beats a clever nested one-liner.

Key syntax

-- chained CTEs: per-channel totals, then above-average channels
WITH per_channel AS (
  SELECT channel, SUM(budget) AS spend
  FROM campaigns GROUP BY channel
),
avg_spend AS (SELECT AVG(spend) AS a FROM per_channel)
SELECT p.channel, p.spend
FROM per_channel p, avg_spend
WHERE p.spend > avg_spend.a;

DAY 6

Rank & dedupe — window functions I

~4 hrs

ROW_NUMBER · RANK · DENSE_RANK · PARTITION BY · dedup · top-N per group.

What you're learning

Window functions compute across a set of rows without collapsing them — the key difference from GROUP BY. Syntax: func() OVER (PARTITION BY ... ORDER BY ...). ROW_NUMBER gives 1,2,3 with no ties; RANK leaves gaps after ties (1,1,3); DENSE_RANK doesn't (1,1,2). The classic uses: dedup (keep the latest row per key with ROW_NUMBER … = 1) and top-N per group.

Key syntax

-- top 3 budget campaigns within each channel
WITH ranked AS (
  SELECT channel, name, budget,
         ROW_NUMBER() OVER (PARTITION BY channel ORDER BY budget DESC) AS rn
  FROM campaigns
)
SELECT channel, name, budget FROM ranked WHERE rn <= 3;

DAY 7

Track trends over time — window functions II

~4 hrs

LAG / LEAD · running totals · moving averages · frames · period-over-period.

What you're learning

LAG(x) / LEAD(x) pull the previous/next row's value — perfect for month-over-month change. Running totals and moving averages use the frame clause: SUM(x) OVER (ORDER BY d ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) for a running total; ROWS BETWEEN 6 PRECEDING AND CURRENT ROW for a 7-row moving average. Growth % = (x - LAG(x)) / LAG(x).

Key syntax

-- month-over-month revenue growth
WITH m AS (
  SELECT DATE_TRUNC('month', order_date) AS mth, SUM(amount) AS rev
  FROM orders GROUP BY 1
)
SELECT mth, rev,
       rev - LAG(rev) OVER (ORDER BY mth) AS mom_change,
       ROUND(100.0 * (rev - LAG(rev) OVER (ORDER BY mth))
             / NULLIF(LAG(rev) OVER (ORDER BY mth), 0), 1) AS mom_pct
FROM m ORDER BY mth;

DAY 8

Build funnels & cohorts — analytics patterns

~4 hrs

Funnels · cohort/retention · date bucketing · A/B comparison.

What you're learning

These are the scenario shapes real analytics interviews and take-homes use, built by combining what you've learned. A funnel counts distinct users reaching each step (visit → signup → activate → pay). Cohort retention groups users by signup month, then measures how many are active N months later. Date bucketing with DATE_TRUNC rolls events into day/week/month. An A/B comparison is a metric grouped by variant.

Key syntax

-- a simple funnel with conditional aggregation
SELECT
  COUNT(DISTINCT lead_id)                                        AS leads,
  COUNT(DISTINCT CASE WHEN event='demo_booked' THEN lead_id END) AS demos,
  COUNT(DISTINCT CASE WHEN status='won'       THEN lead_id END) AS won
FROM lead_events;

DAY 9

Speak every dialect + performance

~3.5 hrs

Snowflake / Oracle / SQL Server differences · indexing · EXPLAIN.

What you're learning

The logic is identical across platforms; only spellings differ. Row limiting: LIMIT n (Postgres/Snowflake/MySQL), SELECT TOP n (SQL Server), FETCH FIRST n ROWS ONLY (Oracle). Snowflake's QUALIFY filters window results without a subquery. Dates differ too (DATEADD vs DATE_ADD vs interval math). On performance: an index is a lookup structure that avoids full scans; EXPLAIN shows the plan; filter early, avoid SELECT *, and keep filters "sargable" (don't wrap the indexed column in a function).

Key syntax

-- "latest row per key" — Snowflake QUALIFY (no subquery)
SELECT *
FROM events
QUALIFY ROW_NUMBER() OVER (PARTITION BY user_id ORDER BY ts DESC) = 1;
